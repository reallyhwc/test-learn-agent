# 05 测试规范

## Java 后端测试

### 测试框架

- Spring Boot Test + MockMvc + AssertJ
- `@SpringBootTest` + `@AutoConfigureMockMvc` 用于 Controller 测试
- `@SpringBootTest` + `@ActiveProfiles("test")` 用于 Service 测试

### 命名约定

- 测试类：`{被测类}Test.java`
- 测试方法：`should{预期行为}`
- 示例：`shouldCreateTransaction`、`shouldFilterByCategory`

### 状态码约定

- POST 创建：期望 `201 Created`（Controller 标注 `@ResponseStatus(HttpStatus.CREATED)`）
- GET 查询：期望 `200 OK`
- 参数校验失败：期望 `400 Bad Request`
- 服务器错误：期望 `500 Internal Server Error`

### 测试数据

- 目录：`src/test/resources/test-data/`
- 使用 `TestDataConfig` 管理测试数据的备份和重置
- `@BeforeEach` 中调用 `testDataConfig.resetAll()` 保证隔离

### 断言风格

- 优先使用 AssertJ 的 `assertThat` 链式断言
- MockMvc 使用 `jsonPath` 验证 JSON 响应
- 示例：`assertThat(balance).isEqualByComparingTo(new BigDecimal("800.00"))`

### 覆盖场景

- 正常流程（happy path）
- 边界值（空值、零值、极大值）
- 异常情况（参数缺失、类型错误）
- 数据隔离（多用户场景）

## 前端测试

### 测试框架

- Vitest + jsdom + @vue/test-utils
- `globals: true`（无需逐文件 import describe/it/expect）
- Element Plus 通过 `config.global.plugins` 注入

### 文件路径与命名

- 测试目录：`finance-frontend/test/`
- 子目录结构映射源码：`test/{components,stores,utils}/*.test.js`
- 命名：`{被测模块}.test.js`

### 组件测试规范

- 使用 `mount()` 或 `shallowMount()` 挂载
- 依赖外部服务的组件使用 `vi.mock()` 模拟
- Element Plus 组件通过 `config.global.stubs` 或真实插件注入
- 异步操作后使用 `await nextTick()` 或 `await flushPromises()`

### Mock 规范

- 函数 mock：`vi.fn()`
- 模块 mock：`vi.mock('模块路径', () => ({ ... }))`
- 全局对象：`vi.stubGlobal('fetch', vi.fn())`
- localStorage：`vi.stubGlobal('localStorage', { getItem: vi.fn(), setItem: vi.fn() })`
- 每个测试后 `vi.restoreAllMocks()` 清理

### SSE 流式测试

- 必须同时覆盖 LF（`\n`）和 CRLF（`\r\n`）两种行尾格式
- 覆盖 `data:xxx`（无空格）和 `data: xxx`（有空格）两种 payload 格式
- 覆盖 event 类型：`data`（默认）、`error`、`thinking`

### 运行命令

```bash
npx vitest run          # 单次运行
npx vitest              # 监听模式
npx vitest run --ui     # UI 界面
```

## Python 测试

### 测试框架

- pytest + pytest-asyncio + httpx
- `asyncio_mode = "auto"`（异步测试自动识别）
- 安装：`pip install -e ".[dev]"`

### 文件路径与命名

- 测试目录：`tests/`
- 命名：`test_{模块名}.py`
- 测试方法：`test_{预期行为}`
- 示例：`test_env_overrides_yaml`、`test_append_and_get`

### Mock 规范

- `unittest.mock.patch` / `unittest.mock.AsyncMock` 用于依赖替换
- `pytest.fixture` 用于共享测试数据
- `tmp_path` fixture 用于文件操作测试（自动清理）
- `monkeypatch.setenv()` 用于环境变量测试

### 异步测试

- 直接 `async def test_xxx()` 即可（`asyncio_mode = "auto"`）
- 无需 `@pytest.mark.asyncio` 装饰器

### 运行命令

```bash
python -m pytest tests/ -v      # 详细输出
python -m pytest tests/ -x      # 遇到失败立即停止
python -m pytest tests/ --tb=short  # 简短 traceback
```

## 通用覆盖率要求

### 必须覆盖的场景矩阵

| 层级 | happy path | 参数校验 | 异常/边界 | 数据隔离 |
|------|:---------:|:-------:|:--------:|:-------:|
| Controller / API 路由 | ✅ 必须 | ✅ 必须 | ✅ 必须 | ✅ 多用户 |
| Service / 核心逻辑 | ✅ 必须 | ✅ 必须 | ✅ 必须 | ✅ 多用户 |
| Repository / 数据层 | ✅ 必须 | - | ✅ 空数据 | ✅ 多用户 |
| 工具类 / 纯函数 | ✅ 必须 | - | ✅ null/空/极值 | - |
| 组件 / UI | ✅ 渲染 | ✅ 表单 | ✅ 空状态 | - |

### 新增功能测试要求

- 新增 Controller/API 端点：必须同步新增对应测试
- 新增 Service 方法：必须同步新增对应测试
- 新增工具函数：必须同步新增对应测试
- 修复 Bug：必须新增覆盖该 Bug 的回归测试

### 禁止行为

- 禁止提交无测试的新 Controller 端点
- 禁止跳过失败的测试（`@Disabled` / `skip` 需附带原因和 Issue 编号）
- 禁止在测试中使用 `Thread.sleep()` 等待（使用 `Awaitility` 或轮询）
